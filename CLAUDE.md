## What this is

Source for the **Quarto book** "Understanding Statistics Through Linear Models" by Dan MacLean ‚Äî the handbook for the TSL Bioinformatics Team "Intro to Stats" course. It teaches statistics to biologists by framing common tests (t-test, ANOVA, chi-square, log-linear, GLMs) as variations on the linear model.

Published site: https://danmaclean.github.io/intro_to_stats/

## Development process (important ‚Äî this is live, mission-critical material)

The book is a published, in-use course handbook. Treat the current content as mission-critical.

- **Never commit changes directly to `master`.** Make changes on a dedicated feature branch and merge via a reviewed PR. `master` should always be in a known-good, renderable state.
- **Render before merging.** A change isn't done until the affected chapter(s) render without R errors (`quarto render <chapter>.qmd`, or a full `quarto render`).
- Keep edits scoped and reversible; prefer small branches over large sweeping ones.

## Build & preview

This is an R + Quarto project using `renv` for R package isolation.

```sh
quarto render           # build the whole book into docs/ (the output-dir)
quarto preview          # live-reload preview while editing
quarto render 03-ttest.qmd   # render a single chapter for a fast check
```

There are no tests or linters ‚Äî "correctness" means the chapters render without R errors and the prose/stats are right. After editing a `.qmd`, render at least that chapter to confirm the R code chunks still execute.

`renv` activates automatically via `.Rprofile`. Note `renv.lock` only pins `renv` itself; the actual analysis dependencies are **not** locked here (see below).

## Architecture / things that aren't obvious

- **Chapter order and structure live in `_quarto.yml`**, not in filenames alone. It defines the parts ("Background", "Common statistical tests", etc.), the chapter sequence, and the appendices. Add or reorder chapters there, not just by creating files.

- **`docs/` is committed build output**, served by GitHub Pages. Don't hand-edit anything under `docs/` ‚Äî it is regenerated by `quarto render`. The `.pdf` and `search.json` there are also generated.

- **The `itssl` package is the heart of the book.** Almost every R chunk calls helper functions named `its_*_time()` (e.g. `its_plot_xy_time`, `its_axplusb_time`, `its_table_time`, `its_mendel_count_data_time`). These live in the external `danmaclean/itssl` package, **not in this repo** ‚Äî install via `devtools::install_github("danmaclean/itssl")`. The functions wrap data generation, plotting, and exercises so the prose can stay focused on concepts. If a chunk references an `its_*` function you can't find, it is defined in that package, not here.

- **Chapter prose convention:** each chapter opens with a numbered list of *Questions / Objectives / Keypoints* before the first `##` section. Match this when adding a chapter.

- **`data/`** holds the few CSVs loaded directly in chapters (e.g. `Interactions_Categorical.csv`, `sample.csv`); `fig/` holds static images referenced from prose. Most figures are generated by R chunks at render time, not stored.

- `09-glms` exists as both `.qmd` and `.ipynb`; the `.qmd` is the one wired into `_quarto.yml`.

- Comments via Hypothes.is are enabled site-wide (`comments: hypothesis` in `_quarto.yml`).

## R code style (deliberate ‚Äî this is a teaching book)

The code is intentionally kept to **"very obvious base R" plus a curated slice of the tidyverse**. Readability for a beginner beats conciseness. When writing or editing R chunks, match these choices:

- **Assignment is always `<-`.** Use `=` only for function arguments. Never assign with `=`.
- **Pipe is always `%>%`** (magrittr), never the native `|>`. `library(dplyr)`/`library(ggplot2)` etc. are loaded before piping.
- **Linear models are the unifying framework.** Tests are taught as `lm(y ~ x, data = df)` inspected with `summary()`. Explicit formula syntax (`y ~ x`, `y ~ a + b + a:b`) is a core teaching metaphor ‚Äî prefer spelling out interactions with `:` over the `*` shorthand. `t.test()`, `aov()`, `wilcox.test()`, `kruskal.test()`, `chisq.test()` appear as the familiar named tests, shown as equivalents to the linear-model view.
- **Tidyverse is used for data wrangling and plotting, base R for the conceptual backbone.** dplyr verbs (`mutate`, `filter`, `group_by`, `summarize`, `if_else`), `readr::read_csv` (explicitly taught over `read.csv`), `tibble::tibble`, `tidyr::pivot_*`, and ggplot2 are all welcome. Base idioms `c()`, `data.frame()`, `$`, `factor()`/`relevel()`, `table()`/`xtabs()` are also standard.
- **The `its_*_time()` helpers exist to hide non-statistical code.** Use them for data generation and routine plotting so chunks stay focused on the statistical point, rather than inlining long ggplot/data-munging boilerplate. Don't reimplement what a helper already does.
- **Deliberately avoided** to keep things beginner-friendly: the native `|>` pipe, the `apply` family (use dplyr verbs), base `[ ]` indexing (prefer `$` or verbs), base `ifelse` (use `if_else`), and model-comparison machinery (`AIC`/`BIC`/`anova()`). GLMs are only touched at the very end.
- **Naming:** `snake_case` for real variables (`mean_two_groups`, `close_fit_model`); single letters (`x`, `y`, `df`, `p`) only for throwaway/illustrative values. Use `pkg::fn()` occasionally to make the source package explicit when it aids the lesson.

## Prose / writing style (deliberate ‚Äî anxious biologists are the audience)

The author writes as a knowledgeable peer-mentor to biologists who find statistics intimidating. The voice is "yes, statistics is hard, *and* you can understand it." Match these conventions when writing or editing prose:

- **Voice:** inclusive **"we"** and direct **"you"**, conversational, contractions throughout (`we'll`, `isn't`, `don't`). Knowledgeable but never stiff or condescending. Light humour is part of the brand (the "~~sadistics~~ statistics" running joke, asides like "By the sweet warmth of the first suns of spring, no.") ‚Äî playful, never flippant.
- **Reassurance is a feature.** Explicitly tell readers to relax about anxiety-inducing topics (test selection, normality assumptions): "Most statistical tests are pretty robust against violations", "Rest assured, the problem isn't insurmountable." Reframe limitations as manageable rather than scary.
- **Intuition before formalism, picture before equation.** Lead with an analogy or a plot, then introduce notation. Keep math notation brief and always follow `$...$` with a plain-English explanation. Deeper math goes in optional "Extra credit" / callout asides the reader can skip ‚Äî the book deliberately avoids proofs and heavy notation ("we're working hard here to avoid using too much mathematical notation").
- **The straight-line / linear-model metaphor is the spine.** New tests are framed as the same idea (a slope = a difference) rather than as separate recipes. Reinforce this thread; the author intentionally repeats it ("I like hammering this point home").
- **Plain-English over jargon.** Introduce a technical term in *italics* and give 2‚Äì3 everyday synonyms ("a Null Model, or perhaps you can think of it as a random model or a default model"). Use concrete, relatable datasets (plants, chickwts, flower colour, voting), and label toy data as toy.
- **Chapter opener (required):** every chapter starts with a numbered **Questions / Objectives / Keypoints** list before the first `##`. Chapters typically end with a **Roundup** summary in a callout.
- **Headings** are sentence case and often full statements of the point ("Straight line relationships are described using two parameters").
- **Callouts** are used sparingly and by purpose: `callout-note` for asides/optional depth, `callout-important` for misconception-correcting essentials, `callout-tip` (often titled "For you to do") for exercises.
- **Formatting:** backticks for R functions/objects/columns/packages (`lm()`, the `group` column, `itssl`); *italics* for new terms and light emphasis; **bold** rarely and only for a critical warning. Blockquotes for the central question driving a section.
- **Signposting phrases** to lean on: "Let's work through‚Ä¶", "Now that we've‚Ä¶", "Recall‚Ä¶", "That's it!", "There you have it". Self-aware meta-commentary (flagging optional material, admitting deliberate repetition) is in keeping with the voice.
